# 1.1 Claude Code Skills

## Supercharge Your Software Engineering Productivity

---

## Introduction

Artificial Intelligence has transformed software development, but simply using an AI coding assistant is no longer enough to maximise productivity. Many developers treat AI as an advanced code generator that produces snippets on demand. While this saves time, it barely scratches the surface of what modern AI-assisted development can achieve.

Claude Code introduces a concept called <span style={{color: '#fb923c', fontWeight: 'bold'}}>Skills — reusable workflows</span> that change how AI participates in software engineering. Instead of merely responding to prompts, <span style={{color: '#fb923c', fontWeight: 'bold'}}>Skills teach Claude structured thinking processes and domain-specific best practices.</span> Rather than generating random code, <span style={{color: '#fb923c', fontWeight: 'bold'}}>Claude begins to work like an experienced software engineer</span> following established methodologies.

> Using Claude Code without Skills means utilising less than 10% of its true capabilities. Skills transform it from a simple coding assistant into an intelligent workflow partner capable of planning, reviewing, testing, designing, and even creating new workflows.

---

## What Are Claude Code Skills?

A Claude Code Skill is a reusable workflow stored inside a folder containing a **`skills.md`** file.

Instead of repeatedly explaining your preferred process, you encode it once into a Skill. Claude then executes that workflow whenever appropriate.

**A Skill contains:**

- YAML front matter
- A name
- A description
- Step-by-step workflow instructions
- Optional reference files (documentation, examples, style guides, internal standards)

The description tells Claude *when* the Skill should activate. The body explains *how* to perform the workflow. Reference materials are loaded only when necessary — keeping context lean.

This design lets Claude follow consistent engineering practices rather than improvising every time.

---

## Installing Official Skills

Official Anthropic Skills can be installed using:

```bash
npx skills add <skill-name>
```

Add the `--global` flag to make the Skill available across every project, not just the current one:

```bash
npx skills add <skill-name> --global
```

---

## Installing Community Skills

Besides official Skills, there is a rich ecosystem of community-created Skills. A community repository indexes over one million Skills.

To install one:

```bash
npx skills add <community-skill-name>
```

The skill name comes from the community listing. This means developers don't have to reinvent workflows that others have already formalised.

## Skill 1 — Front-End Design Skill

### The Problem

AI-generated user interfaces often look artificial. Common symptoms:

- Generic spacing with no visual rhythm
- Unnecessary hover effects everywhere
- Poor visual hierarchy — everything looks equally important
- Overused gradients
- Layouts that feel assembled, not designed

### How the Skill Solves This

The Front-End Design Skill embeds actual design principles into Claude's workflow. Instead of randomly arranging components, it considers:

- **Design tokens** — spacing scales, colour palettes, type sizes
- **Layout rules** — grid systems, alignment, whitespace
- **Visual hierarchy** — contrast, weight, size relationships
- **Interface consistency** — reusing established patterns
- **Human-centred design principles** — what the user actually needs to see first

The result is an interface that appears intentionally designed rather than automatically generated.

## Plan Mode

Before writing a single line of code, Claude offers **Plan Mode**.

Instead of immediately modifying files, it:

1. Reads project context
2. Understands existing architecture
3. Asks clarifying questions
4. Builds an implementation strategy
5. Waits for your approval before touching anything

**Key insight:** Plan Mode is read-only. No files change until you approve the plan.

This eliminates the most common AI coding mistake — confidently implementing the wrong thing. Alignment happens before implementation, not during a painful rollback.

---

## Auto Mode

Normally AI assistants request permission repeatedly when accessing files. **Auto Mode** streamlines this.

It:

- Automatically performs safe, low-risk operations
- Pauses for risky or irreversible actions
- Reduces unnecessary interruptions

This creates a smoother development experience while keeping the safety net in place. Think of it like giving a trusted engineer access to the codebase — they don't ask permission to read a file, but they do ask before deleting one.

## Desktop Preview

Instead of imagining what the interface looks like, Claude can:

- Launch the application
- Open a live preview
- Inspect the rendered output
- Take screenshots
- Analyse the DOM

This creates a real feedback loop. The AI observes the actual UI and refines it accordingly — the same way a developer would in a browser dev tools session.

---

## Checkpoints

One common frustration with AI coding: you had a working implementation, asked for one more improvement, and now something is broken.

Claude automatically creates **checkpoints** — snapshots of the project state before significant changes.

If later modifications make things worse, you revert to an earlier checkpoint instead of manually recovering lost work. Think of it as automatic git stashing at every meaningful step.

## Skill 2 — Grow with Docs

This Skill enforces a discipline that separates senior engineers from junior ones: **understand before you build**.

Instead of immediately implementing features, Claude interrogates you about requirements and explores the existing codebase first:

- Existing documentation
- Domain models
- Architecture decision records
- Internal naming conventions
- Established terminology

### Why This Matters

Large projects have accumulated context. If you propose something inconsistent with the existing architecture, the Skill surfaces that conflict early — during planning, not three weeks into implementation.

### Example Workflow

Adding a new plugin to Kibana. Before asking implementation questions, Claude:

1. Searches the repository
2. Reads source files
3. Explores existing plugins
4. Uses multiple agents to inspect different sections in parallel
5. Builds a context document

*Only then* does it begin asking the developer questions. The resulting architectural decisions are dramatically better because they're grounded in what already exists.

## Skill 3 — TDD Workflow

**TDD = Test-Driven Development.** The classic three-step loop:

1. Write a failing test
2. Write implementation code to make it pass
3. Refactor

### The Common AI Problem

Without structured guidance, AI typically:

- Writes implementation first
- Creates tests afterward
- Produces tests that simply validate its own code

Such tests rarely catch real defects — they're just mirrors of the implementation.

### How the Skill Enforces Proper TDD

Using the example of adding a Share button:

**Step 1 — Generate tests.** Write the test cases before any implementation exists.

**Step 2 — Generate edge-case tests.** What happens with empty inputs, null values, concurrent users, network failures?

**Step 3 — Run tests.** They fail. This is correct — the implementation doesn't exist yet.

**Step 4 — Write implementation.** Only now does Claude write the feature code.

**Step 5 — Run tests again.** Now they pass.

**Step 6 — Measure coverage.** The Skill enforces a minimum threshold (typically 80%+).

This workflow creates stronger regression protection and catches bugs that implementation-first development misses entirely.

## Skill 4 — Code Reviewer

This Skill acts like a senior engineer reviewing a pull request — not just checking syntax, but evaluating:

- **Security** — are there injection risks, authentication gaps, unsafe defaults?
- **Maintainability** — can a future engineer understand this in six months?
- **Architecture** — does this fit the existing design, or does it fight it?
- **Testing** — are edge cases covered? Is coverage meaningful?
- **Naming** — do identifiers communicate intent?
- **Reliability** — what breaks under load or partial failure?

### Example Review — Kibana CSRF Validation PR

**Critical findings:**

Architectural assumptions exist in comments but are not enforced in code. If infrastructure changes, the security protections could silently disappear.

**High priority findings:**

Configuration values lack constraints. Future variations in capitalisation or schema naming could silently break intended behaviour.

**Medium priority findings:**

No debug logging when security checks are bypassed. Such logs are essential for incident response and monitoring.

**Other findings:**

Missing tests, naming issues, maintainability gaps — all surfaced and explained with suggested fixes.

The Skill behaves like an experienced security reviewer on your team, available on every PR.

## Skill 5 — Skill Creator

The most powerful feature: **Claude can create new Skills**.

Instead of manually writing markdown workflows, you describe your process in plain language. Claude then:

1. Interviews you about the workflow
2. Asks clarifying questions
3. Understands the exact requirements
4. Generates the Skill
5. Packages it correctly as a reusable `skills.md` file

### Example — Elasticsearch Query Validator

The desired Skill should:

- Validate DSL syntax
- Detect deprecated query types
- Follow Kibana performance guidance
- Analyse current branch differences against main
- Report findings in Markdown
- Suggest concrete fixes

Claude asks clarification questions, then generates a reusable Skill that any engineer on the team can invoke going forward.

## Why Skills Matter

Skills shift AI from **reactive coding** to **structured engineering**.

| Without Skills | With Skills |
|---|---|
| Prompt → random code | Prompt → defined workflow |
| Inconsistent quality | Repeatable standards |
| Improvised testing | TDD enforced |
| Generic UI output | Design-principled output |
| Ad hoc reviews | Systematic evaluation |
| Manual workflow docs | Automated Skill creation |

Instead of repeatedly prompting the AI with the same instructions session after session, you encode best practices once and reuse them across every project, every engineer, every day.

---

## The Big Picture

- **Skills are reusable workflows**, not one-off prompts — encode your process once, apply it everywhere
- **Plan Mode prevents the most expensive mistake** — building the wrong thing confidently
- **TDD Workflow produces tests that actually catch bugs** — not mirrors of the implementation
- **Code Reviewer surfaces critical issues early** — security, architecture, and maintainability in one pass
- **Grow with Docs enforces understanding before building** — grounding decisions in existing architecture
- **Skill Creator compounds your investment** — your custom methodologies become reusable assets

> **Simple takeaway:** Skills don't make Claude faster at coding — they make Claude smarter about *what* to code and *how* to do it correctly the first time.
