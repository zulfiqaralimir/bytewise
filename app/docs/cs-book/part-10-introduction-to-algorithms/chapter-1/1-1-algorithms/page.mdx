# 1.1 Algorithms

---

## What Is an Algorithm?

**The one-line definition from CLRS:**

> An algorithm is a well-defined computational procedure that takes some input, does a finite amount of work, and produces the correct output.

**Three words do all the work:**

- **Well-defined** — every step is clear and unambiguous. No room for "it depends." A computer needs exact instructions.
- **Finite** — it always stops. It does not run forever. It takes some number of steps and finishes.
- **Correct** — it produces the right answer. Not sometimes. For every possible valid input.

**Think of it like a recipe:**
- Ingredients = input
- Steps = the algorithm
- Dish = output
- A good recipe is unambiguous, always finishes, and always works

## The Sorting Problem — The Book's Running Example

**Why sorting?**
- Simple to understand
- Rich enough to reveal deep trade-offs
- CLRS uses it as a teaching canvas throughout the whole book

**The problem:**
- **Input:** A list of numbers in any order — `[31, 41, 59, 26, 41, 58]`
- **Output:** The same numbers sorted smallest to largest — `[26, 31, 41, 41, 58, 59]`

**What makes it interesting:**
- Dozens of completely different algorithms all solve this correctly
- Some are fast, some are slow, some use more memory, some use less
- Choosing between them is an engineering decision — and it depends on context

**Sorting algorithms you will meet in this book:**
- Insertion sort
- Merge sort
- Heapsort
- Quicksort
- And more — each solving the same problem differently

## Three Terms You Need to Know

### 1. Instance of a Problem

- A **problem** is general — "sort a list of numbers"
- An **instance** is one specific input — "sort `[5, 3, 8, 1]`"

**Why it matters:**
- An algorithm must solve every instance — not just the easy ones
- If it fails on even one valid input, it is not correct

### 2. Correct Algorithm

**An algorithm is correct if:**
- It halts (finishes) for every input
- It produces the right answer for every input — including edge cases

**Edge cases to always think about:**
- An empty list
- A list with one item
- A list already sorted
- A list sorted in reverse

**Think of it like an exam:**
- Getting most questions right is not enough
- Correctness means every question — including the tricky ones at the end

**Why testing alone is not enough:**
- You can test a million inputs and still miss the one that breaks it
- That is why CLRS teaches formal proof techniques alongside algorithms

### 3. Data Structures

**What it is:**
- The way you organize and store data in memory

**Why it matters:**
- Algorithms and data structures are always a pair
- Algorithm = *what to do*
- Data structure = *how the data is arranged*
- Wrong data structure → fast algorithm becomes slow

**Think of it like a kitchen:**
- Recipe = algorithm
- How your ingredients and tools are organized = data structure
- A chef who cannot find anything will be slow — even with a perfect recipe

## Why Efficiency Matters

**Two correct algorithms can be vastly different in practice.**

**Example — searching a phone book:**

| Strategy | How it works | Steps for 1,000 pages |
|---|---|---|
| Flip page by page | Check every page from the start | Up to 1,000 steps |
| Open to the middle | Halve the search space each time | About 10 steps |

Both always find the name. One is 100x faster.

**Now scale to the internet:**
- Google does not search 1,000 pages — it searches billions
- Strategy 1 (flip every page) would take years
- Strategy 2 (halve each time) takes milliseconds

**The lesson:**
- Efficiency is not an optional optimization
- At scale, it is the difference between a system that works and one that does not

**Coming up in Chapter 2:**
- CLRS formalizes this with **running time analysis**
- Tools: Big O notation, best case, worst case, average case
- All of it flows from this simple idea — two correct algorithms can do dramatically different amounts of work

## Real-World Example — Google Search

**The problem Google solves:**
- Return the right result from billions of webpages
- In under one second
- For billions of queries per day

**Why it works — three reasons that map directly to 1.1:**

- **Correctness** — the algorithm must return relevant results every time, not just sometimes
- **Efficiency** — fast enough to serve billions of queries per day without slowing down
- **Data structures** — Google pre-builds an inverted index (a specific data structure) so it never has to read every webpage live

**What would happen without good algorithm selection:**
- A "check every webpage" approach would take hours per search
- No hardware upgrade fixes a fundamentally slow algorithm

> **Why this algorithm?** Google uses inverted indexes with ranking algorithms (like PageRank) because they allow sub-second lookup across billions of documents — something a naive approach could never achieve at any scale.

## HLD & LLD Connections

### In System Design (HLD)

Every major architectural decision has an algorithm underneath it:

| System Component | Algorithm at Work |
|---|---|
| Database index | B-Tree — binary search on sorted structure |
| Cache eviction | LRU / LFU — decides what to remove when cache is full |
| Load balancer | Round-robin, least connections, consistent hashing |
| Message queue | Priority scheduling — decides which message to process next |
| Search engine | Inverted index + ranking algorithm |

**The big point:**
- Algorithm choice at the HLD level determines whether your system handles 1,000 users or 100 million users

### In Low-Level Design (LLD)

Every time you choose a data structure, you are also choosing the algorithms that come with it:

| Data Structure | Fast At | Slow At |
|---|---|---|
| Array | Access by index — O(1) | Search — O(n) |
| Hash Map | Search and insert — O(1) | Ordered access |
| Sorted List | Binary search — O(log n) | Insert — O(n) |
| Linked List | Insert at head — O(1) | Access by index — O(n) |

**The big point:**
- These are design decisions, not implementation details
- A class built on the wrong data structure may need a full redesign later
- Getting this right early is the mark of a senior engineer

> **Simple takeaway:** In HLD, algorithms determine system scale. In LLD, algorithms determine class performance. Both levels matter — and both start with understanding what an algorithm actually is.

## The Big Picture

**Key ideas to carry forward from 1.1:**

- An algorithm is a precise, finite, correct procedure — not a vague plan
- The same problem can have many correct algorithms — correctness is the minimum bar, not the finish line
- Efficiency determines whether a correct algorithm is actually useful at scale
- Data structures and algorithms are inseparable — the choice of one shapes the choice of the other
- Every real system you use today — search, maps, social media, finance — is built on algorithm and data structure decisions made by engineers

> **Simple takeaway:** An algorithm is only as good as two things — whether it is always correct, and whether it is fast enough to be practical. Section 1.1 is the book reminding you that both matter, before it teaches you how to measure either.

## Practice — LeetCode Problems

These problems directly reinforce what 1.1 teaches — the difference between a correct algorithm and an efficient one, and how data structure choice changes everything.

| # | Problem | Difficulty | Why It Fits |
|---|---|---|---|
| 1 | [Two Sum](https://leetcode.com/problems/two-sum/) | 🟢 Easy | A brute-force correct solution exists (nested loop) but a smarter data structure (hash map) makes it 10x faster — the exact correctness vs efficiency trade-off in 1.1 |
| 2 | [Contains Duplicate](https://leetcode.com/problems/contains-duplicate/) | 🟢 Easy | Choosing between array scan vs hash set is a pure data structure decision — wrong choice is correct but slow |
| 3 | [Binary Search](https://leetcode.com/problems/binary-search/) | 🟢 Easy | The phone book analogy from 1.1 in code — halving the search space vs scanning every element |
| 4 | [Search Insert Position](https://leetcode.com/problems/search-insert-position/) | 🟢 Easy | Reinforces what "correct for every instance" means — edge cases like empty arrays and boundary positions |
| 5 | [Missing Number](https://leetcode.com/problems/missing-number/) | 🟢 Easy | Multiple correct algorithms exist — brute force, sorting, hash set, math — perfect for comparing efficiency across solutions |

**How to use these:**

- Solve each problem yourself first
- Then ask: is my solution correct for every possible input — including edge cases?
- Then ask: is there a faster solution that uses a better data structure?
- That two-step thinking is exactly what 1.1 is building

> **Simple rule:** A solution that passes all test cases is correct. A solution that does it efficiently is engineering.
