# 1.1 Algorithms

## What Is an Algorithm?

Here is the one-line definition from CLRS:

> An algorithm is a well-defined computational procedure that takes some input, does a finite amount of work, and produces the correct output.

Three words in that definition do a lot of work: **well-defined**, **finite**, and **correct**.

- **Well-defined** — every step is clear and unambiguous. There is no room for "it depends" or "use your judgment." A computer needs exact instructions.
- **Finite** — the algorithm always stops. It does not run forever. It takes some number of steps and then finishes.
- **Correct** — it produces the right answer. Not sometimes. For every possible valid input.

Think of it like a recipe. You give it ingredients (input), follow the steps, and get a dish (output). A good recipe is unambiguous, always finishes, and always produces something edible. A bad recipe might be vague, loop forever, or occasionally produce something inedible — that is a bad algorithm.

## The Sorting Problem — The Book's Running Example

CLRS uses sorting as its go-to example throughout the book. It is a perfect teaching problem because it is simple to understand but rich enough to reveal deep ideas.

**The problem:**

- **Input:** A list of numbers in any order — say `[31, 41, 59, 26, 41, 58]`
- **Output:** The same numbers rearranged from smallest to largest — `[26, 31, 41, 41, 58, 59]`

Easy to state. But here is what makes it interesting: there are dozens of completely different algorithms that all solve this problem correctly. Some are fast, some are slow, some use a lot of memory, some use very little. The sorting problem exists in the book as a canvas for comparing those trade-offs.

You will encounter many sorting algorithms in this book — insertion sort, merge sort, heapsort, quicksort, and more. Each one solves the same problem. Each one does it differently. Choosing between them is an engineering decision, and that decision depends on context.

## Three Terms You Need to Know

### 1. Instance of a Problem

A **problem** is general. A **instance** is one specific input.

- The *sorting problem* is the general task of putting numbers in order.
- Sorting the list `[5, 3, 8, 1]` is one *instance* of that problem.

This distinction matters because an algorithm must solve every instance — not just the easy ones, not just the ones you tested. If your algorithm fails on even one valid input, it is not correct.

### 2. Correct Algorithm

An algorithm is **correct** if it halts with the right answer for every possible input instance.

Think of it like a student taking an exam. Getting question 1 right is not enough. Getting most questions right is not enough. Correctness means getting every question right, including edge cases — an empty list, a list with one item, a list where everything is already sorted, a list where everything is in reverse.

This is why testing is never enough to prove correctness. You can test a million inputs and still miss the one that breaks your algorithm. That is why CLRS teaches formal proof techniques alongside the algorithms themselves.

### 3. Data Structures

A **data structure** is the way you organize and store data in memory.

Algorithms and data structures are always a pair. The algorithm describes *what to do*. The data structure describes *how the data is arranged*. Choosing the wrong data structure can make a fast algorithm slow, or make an easy problem hard.

Think of it like a kitchen. The recipe (algorithm) tells you how to cook. The way you organize your ingredients and tools (data structure) determines how efficiently you can follow that recipe. A chef who cannot find anything in their kitchen will be slow even if they know the recipe perfectly.

## Why Efficiency Matters

Two algorithms can both be correct and still be vastly different in practice.

Imagine searching a phone book:

- **Strategy 1:** Flip through every page from the beginning until you find the name. Always works. Very slow.
- **Strategy 2:** Open to the middle, decide whether the name is in the left half or right half, repeat. Always works. Much faster.

Both are correct. But Strategy 1 might take 1,000 steps on a 1,000-page phone book. Strategy 2 takes about 10 steps on the same book — because each step cuts the remaining search in half.

Now scale that to the internet. Google does not search a 1,000-page phone book. It searches billions of pages. Strategy 1 would take years. Strategy 2 takes milliseconds.

This is why efficiency is not an optional optimization — it is often the difference between a system that works in the real world and one that does not.

CLRS formalizes this with **running time analysis**, which you will get deep into starting in Chapter 2. The tools for measuring efficiency — Big O notation, best case, worst case, average case — all flow from this simple idea: two correct algorithms can be dramatically different in the amount of work they do.

## Where You Will See This

Algorithms are everywhere once you start looking. Every piece of software you use today is running algorithms underneath:

- **Search** — Google, Bing, and database queries all use algorithms to find the right result from billions of options
- **Sorting** — spreadsheets, leaderboards, ranked search results, your email inbox sorted by date
- **Navigation** — Google Maps, GPS routing, and network packet routing all use path-finding algorithms
- **Finance** — stock exchange matching engines pair buyers and sellers using algorithms in microseconds
- **Social media** — your feed, your recommendations, who you might know — all algorithm-driven decisions
- **AI and machine learning** — every model trains and predicts using algorithms at its core

The concepts in 1.1 — correctness, efficiency, the right data structure — show up in every single one of these.

## Real-World Example

### How Google Search Decides What to Show You

When you type a search query, Google does not read every webpage on the internet to find your answer. That would take hours. Instead, it runs a series of carefully chosen algorithms on pre-built data structures — indexes — that were constructed in advance.

The reason Google returns results in under a second is not raw hardware power alone. It is **algorithm selection**. Google's engineers chose algorithms that can search billions of documents in logarithmic time — meaning doubling the number of webpages barely increases the search time at all.

This is a direct real-world consequence of what 1.1 is teaching:

- **Correctness** — the algorithm must return relevant results every time, not just sometimes
- **Efficiency** — the algorithm must be fast enough to serve billions of queries per day
- **Data structures** — the inverted index (a specific data structure) is what makes fast lookup possible in the first place

Change any one of those three and Google either returns wrong results, takes too long, or crashes under load. All three have to be right simultaneously.

> **Why this algorithm?** Google uses a combination of inverted indexes and ranking algorithms (like PageRank) because they allow sub-second lookup across billions of documents — something a naive "check every page" approach could never achieve at that scale.

## HLD & LLD Connections

### In System Design (HLD)

Every major architectural decision in system design has an algorithm underneath it. When you design a large system, you are constantly making algorithm choices — even when you do not call them that.

- **Databases** — when you add an index to a database column, the database builds a sorted data structure (usually a B-Tree) that uses binary search for lookups. The algorithm makes queries fast.
- **Caching systems** — deciding which data to evict from a cache (LRU, LFU) is an algorithm problem
- **Load balancers** — distributing traffic across servers uses algorithms (round-robin, least connections, consistent hashing)
- **Message queues** — prioritizing which messages to process first is a sorting and scheduling algorithm problem

At the HLD level, algorithm choice determines whether your system can handle 1,000 users or 100 million users.

### In Low-Level Design (LLD)

At the code and class design level, every time you choose a data structure, you are also choosing the algorithms that come with it.

- Choosing an **array** means search is slow (scan everything) but access by index is instant
- Choosing a **hash map** means search is fast but order is lost
- Choosing a **sorted list** means you can binary search but inserts are slower

These are not implementation details — they are design decisions. A class you design today with the wrong internal data structure might be 100x slower than it needs to be, and the fix will require a full redesign later.

> **Simple takeaway:** In HLD, algorithms determine system scale. In LLD, algorithms determine class performance. Understanding both levels is what separates a junior engineer from a senior one.

## The Big Picture

Section 1.1 is not throwing algorithms at you yet. It is building your mental model before any code or math appears.

The key ideas to carry forward:

- An algorithm is a precise, finite, correct procedure — not a vague plan
- The same problem can have many correct algorithms — correctness is the minimum bar, not the finish line
- Efficiency determines whether a correct algorithm is actually useful at scale
- Data structures and algorithms are inseparable — the choice of one shapes the choice of the other

> **Simple takeaway:** An algorithm is only as good as two things — whether it is always correct, and whether it is fast enough to be practical. Section 1.1 is the book reminding you that both matter, before it teaches you how to measure either.
