# ✅ Function Calls (Step-by-Step)

## Lesson 18: What the CPU Actually Does When You Call a Function

---

# 🎯 Learning Objective

By the end of this lesson, I should be able to answer:

* What is a Function Call?
* Why do functions exist?
* What happens when a function is called?
* How does the CPU execute a function call?
* What is a Stack Frame?
* How are parameters passed?
* How are return values returned?
* Why is the Stack important for function calls?
* What would happen if function calls did not exist?

---

# 🌍 The Fundamental Problem

Imagine writing:

```c
printf("Hello");
printf("Hello");
printf("Hello");
printf("Hello");
printf("Hello");
```

Suppose the same code is needed:

```text
10 Times

100 Times

1000 Times
```

Writing it repeatedly is inefficient.

Engineers invented:

# Functions

A function allows us to:

```text
Write Once

Use Many Times
```

---

## What is a Function?

A function is:

> A reusable block of code that performs a specific task.

Example:

```c
void greet()
{
    printf("Hello");
}
```

---

## Function Call

Using the function:

```c
greet();
```

This is called a:

# Function Call

---

## Real-Life Analogy

Imagine a restaurant.

You:

```text
Place Order
```

Kitchen:

```text
Prepares Food
```

Food:

```text
Returned To You
```

Similarly:

```text
Call Function

↓

Function Executes

↓

Returns Control
```

---

## Example

```c
void greet()
{
    printf("Hello");
}

int main()
{
    greet();
}
```

Execution:

```text
main()

↓

greet()

↓

Print Hello

↓

Return To main()
```

---

## What Actually Happens?

Most beginners think:

```text
greet();

↓

Magic
```

No.

The CPU performs many steps.

---

## Step 1: Function Call Instruction

Suppose:

```c
greet();
```

Compiler may generate:

```assembly
CALL greet
```

Meaning:

```text
Jump To greet()

But Remember

Where To Return
```

---

## Why Remember?

After:

```c
greet();
```

execution must continue at:

```c
printf("Done");
```

Example:

```c
greet();

printf("Done");
```

CPU must remember:

```text
Where execution should continue.
```

---

## Return Address

Before jumping:

CPU stores:

```text
Address Of Next Instruction
```

Example:

```text
100 → CALL greet

101 → printf("Done")
```

CPU saves:

```text
101
```

This is called:

# Return Address

---

## Where Is Return Address Stored?

On the:

# Stack

Example:

```text
Stack

+----------------+
| Return Address |
+----------------+
```

---

## Step 2: Create Stack Frame

Function:

```c
void greet()
{
    int x = 10;
}
```

CPU creates:

# Stack Frame

A Stack Frame stores:

```text
Parameters

Local Variables

Return Address

Saved Registers
```

---

## Example

```c
void greet()
{
    int x = 10;
}
```

Stack:

```text
+----------------+
| Return Address |
+----------------+
| x = 10         |
+----------------+
```

---

## Why Stack Frames Exist

Imagine:

```c
funcA()
{
    funcB();
}
```

and:

```c
funcB()
{
}
```

CPU must remember:

```text
Which Function Called Which
```

Stack Frames solve this problem.

---

## Example

```text
main()

↓

funcA()

↓

funcB()
```

Stack:

```text
+-----------+
| funcB     |
+-----------+
| funcA     |
+-----------+
| main      |
+-----------+
```

---

## Step 3: Execute Function

After Stack Frame creation:

CPU executes instructions inside:

```c
greet()
```

Example:

```c
void greet()
{
    printf("Hello");
}
```

CPU executes:

```text
Print Hello
```

---

## Step 4: Return Statement

Example:

```c
return;
```

or

```c
return 25;
```

Function finishes.

---

## Step 5: Remove Stack Frame

CPU removes:

```text
Local Variables

Parameters

Temporary Data
```

from the stack.

Example:

Before:

```text
+----------------+
| Return Address |
+----------------+
| x              |
+----------------+
```

After:

```text
Removed
```

---

## Step 6: Return To Caller

CPU uses:

```text
Return Address
```

stored earlier.

Example:

```text
Return Address = 101
```

CPU jumps back:

```text
Instruction 101
```

Program continues.

---

## Complete Function Call Flow

```text
CALL Function

↓

Store Return Address

↓

Create Stack Frame

↓

Execute Function

↓

Remove Stack Frame

↓

Return To Caller
```

This happens every function call.

---

## Parameters

Example:

```c
void add(int a, int b)
{
}
```

Call:

```c
add(10, 20);
```

Parameters:

```text
a = 10

b = 20
```

must be passed to the function.

---

## Stack Example

```text
+----------------+
| Return Address |
+----------------+
| a = 10         |
+----------------+
| b = 20         |
+----------------+
```

Function accesses:

```text
a

b
```

from its Stack Frame.

---

## Return Values

Example:

```c
int add()
{
    return 50;
}
```

Question:

> How does 50 return to the caller?

---

## Simple View

CPU places:

```text
50
```

into a special register.

Then returns.

Example:

```text
Return Register

↓

50
```

Caller reads it.

---

## Example

```c
int result = add();
```

Execution:

```text
add()

↓

Return 50

↓

Special Register

↓

result = 50
```

---

## Nested Function Calls

Example:

```text
main()

↓

funcA()

↓

funcB()

↓

funcC()
```

Stack:

```text
+-----------+
| funcC     |
+-----------+
| funcB     |
+-----------+
| funcA     |
+-----------+
| main      |
+-----------+
```

---

## Returning

Functions return in reverse order:

```text
funcC

↓

funcB

↓

funcA

↓

main
```

This is why Stack uses:

# LIFO

Last In First Out

---

## Recursion

Example:

```c
void func()
{
    func();
}
```

Every call creates:

```text
New Stack Frame
```

Example:

```text
func

func

func

func

func
```

Stack keeps growing.

Eventually:

```text
Stack Full
```

Result:

# Stack Overflow

---

## Why Function Calls Are Important

Without function calls:

Programs would contain:

```text
Huge Repeated Code
```

making software:

```text
Hard To Read

Hard To Maintain

Hard To Reuse
```

Functions solve this.

---

## Function Calls and C Programming

Almost everything in C relies on function calls:

```c
printf()

scanf()

malloc()

free()

fopen()
```

All are functions.

---

## Visual Summary

```text
main()

↓

CALL greet()

↓

Save Return Address

↓

Create Stack Frame

↓

Execute greet()

↓

Destroy Stack Frame

↓

Return To main()
```

---

## Common Misconceptions

❌ Function calls create new processes.

❌ Local variables are stored on the Heap.

❌ Return addresses are stored in RAM randomly.

❌ Function calls happen without using the Stack.

❌ Returning from a function means restarting the program.

---

# 🧠 Quick Revision

* Function = reusable block of code.
* Function Call = invoking that block.
* CPU performs:

  * Save Return Address
  * Create Stack Frame
  * Execute Function
  * Destroy Stack Frame
  * Return To Caller
* Stack Frames contain:

  * Parameters
  * Local Variables
  * Return Address
  * Saved Registers
* Return values are usually returned through registers.
* Function calls rely heavily on the Stack.
* Recursion creates multiple Stack Frames.
* Too many Stack Frames can cause Stack Overflow.

# ✅ What is the Runtime? (And Why Does main() Need It?)

# 🎯 The Question

You learned:

```text
Operating System

↓

main()
```

But I also said:

```text
Operating System

↓

Runtime

↓

main()
```

Question:

> Why doesn't the OS directly call main()?

Because before `main()` can run, many things must be prepared.

The component that prepares everything is called:

# Runtime

---

## What is Runtime?

Runtime is a small piece of code that executes before your `main()` function.

Its job is:

```text
Prepare Program Environment

↓

Call main()

↓

Cleanup After main()
```

Think of it as:

```text
Program Manager
```

that prepares everything before the actual work starts.

---

## Real Life Analogy

Imagine a teacher entering a classroom.

Before teaching starts:

```text
Open Classroom

↓

Turn On Lights

↓

Arrange Chairs

↓

Check Attendance

↓

Start Lecture
```

The lecture is:

```text
main()
```

Everything before the lecture is:

```text
Runtime
```

---

## What Happens When You Run a Program?

Suppose:

```bash
./program
```

You might think:

```text
OS

↓

main()
```

Wrong.

Actual flow:

```text
OS

↓

Load Program

↓

Runtime Startup Code

↓

main()

↓

Runtime Cleanup

↓

Program Ends
```

---

## Why Can't the OS Directly Call main()?

Suppose your program contains:

```c
#include <stdio.h>

int global = 100;

int main()
{
    printf("%d", global);
}
```

Before `main()` starts:

```text
global must already exist
```

Someone must initialize it.

That "someone" is the Runtime.

---

## Runtime Responsibilities

The runtime performs many tasks.

---

## 1. Setup Stack

Functions need:

```text
Stack Frames

Local Variables

Function Calls
```

Before:

```c
int main()
{
}
```

runs, the runtime creates:

```text
Program Stack
```

Without Runtime:

```text
No Stack

↓

No Function Calls

↓

Program Crashes
```

---

## 2. Setup Heap

Functions like `malloc()` need Heap memory.

Before `main()` starts:

```text
Heap Region Must Exist
```

Runtime prepares it.

Without Runtime, `malloc(100)` would have nowhere to allocate memory.

---

## 3. Initialize Global Variables

Example:

```c
int global = 100;
```

Before `main()`:

```text
global must become 100
```

Runtime performs this initialization.

Without Runtime, global variables might contain garbage values.

---

## 4. Initialize Static Variables

Example:

```c
static int counter = 50;
```

Before `main()`:

```text
counter must become 50
```

Runtime handles this.

---

## 5. Prepare Standard Library

Functions:

```c
printf()

scanf()

malloc()

fopen()
```

need internal setup.

Runtime initializes the C library.

Without Runtime, even `printf("Hello")` might fail.

---

## 6. Process Command-Line Arguments

Example:

```c
int main(int argc, char *argv[])
{
}
```

If user runs:

```bash
./program file.txt
```

Runtime creates:

```text
argc = 2

argv[0] = "./program"

argv[1] = "file.txt"
```

before calling `main()`.

Without Runtime, `argc` and `argv` wouldn't exist.

---

## 7. Call main()

Only after everything is ready:

```text
Runtime

↓

Calls main()
```

---

## What Happens When main() Returns?

Example:

```c
int main()
{
    return 0;
}
```

You might think:

```text
Program Ends
```

Not immediately.

Control returns to:

```text
Runtime
```

Runtime performs cleanup:

```text
Close Files

Flush Output Buffers

Release Resources

Return Exit Code To OS
```

---

## Complete Flow

```text
Operating System

↓

Load Program Into RAM

↓

Runtime Startup Code

↓

Initialize Globals

↓

Setup Stack

↓

Setup Heap

↓

Initialize Libraries

↓

Call main()

↓

main() Executes

↓

main() Returns

↓

Runtime Cleanup

↓

Return Status To OS

↓

Program Ends
```

---

## Visual Diagram

```text
+----------------------+
| Operating System     |
+----------------------+
           |
           v
+----------------------+
| Runtime Startup Code |
+----------------------+
           |
           v

| main()               |
+----------------------+
           |
           v

           |
           v
```

## Where is the Runtime Code?

When you compile:

```bash
gcc main.c
```

the compiler automatically links startup files.

Examples on Linux:

```text
crt0.o

crti.o

crtn.o
```

These contain runtime startup code. You usually don't see them.

Even the smallest C program uses runtime code.

---

## Why Systems Programmers Care

Understanding runtime helps explain:

* Program startup
* Function calls
* Stack creation
* Heap creation
* Global variable initialization
* Program termination
* Operating system interaction

---

## The Most Important Insight

Many beginners think:

```text
OS

↓

main()
```

Reality:

```text
OS

↓

Runtime

↓

main()

↓

Runtime

↓

OS
```

`main()` is not the true beginning of execution.

The runtime starts first, prepares the environment, calls `main()`, and cleans up after it returns.

---

# 🧠 Quick Revision

Runtime is responsible for:

* Creating the Stack
* Creating the Heap
* Initializing Global Variables
* Initializing Static Variables
* Preparing Standard Libraries
* Processing Command-Line Arguments
* Calling `main()`
* Cleaning up after `main()`

Without runtime:

```text
No Stack

No Heap

No printf()

No malloc()

No Global Initialization

No main()
```

would work correctly.
