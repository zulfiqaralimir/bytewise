## Level 1: Registers

Registers are inside the CPU.

Examples:

```text

RBX

RCX
```

**Purpose:** Store data currently being processed.

**Advantage:** Fastest memory available.

**Disadvantage:** Extremely small capacity.

---

### Example

```c

```

The CPU tries to place `a`, `b`, and `c` inside registers before performing the addition.

**Purpose:** Store frequently used data close to the CPU so it does not have to wait for RAM.

### Why Multiple Cache Levels?

A large cache is slower and more expensive to build.

Engineers compromise:

```text

+

Larger Slower Cache  (L2, L3)
```

This gives the CPU fast access for the hottest data and a fallback for less-used data.

Data is not found in cache.


## Why Memory Hierarchy Works: Locality of Reference

Most programs repeatedly access a small amount of data.

This principle is called **Locality of Reference**.

It is the reason caches are so effective.

---

## Temporal Locality

If a memory location was used recently, it will likely be used again soon.

Example:

```c
count++;
count++;
count++;
```

The same variable `count` is accessed repeatedly вЂ” it stays in cache.

---

## Spatial Locality

If memory location `100` is used, nearby locations such as `101`, `102`, `103` will likely be used soon.

Example:

```c
array[0]
array[1]
array[2]
```

The CPU loads a block of memory into cache at once, so accessing `array[0]` brings `array[1]`, `array[2]`, etc. along for free.

## Why C Programmers Must Understand Memory Hierarchy

Memory Hierarchy affects:

- Program speed
- Cache efficiency
- Data structure choices
- Algorithm selection
- Operating systems
- Databases
- Game engines

---

### Example вЂ” Cache Locality Matters

These two loops perform the same arithmetic but behave very differently:

**Loop 1 вЂ” Sequential access (cache-friendly):**

```c
for (i = 0; i < n; i++)
{
    sum += array[i];
}
```

**Loop 2 вЂ” Random access (cache-unfriendly):**

```c
for (i = 0; i < n; i++)
{
    sum += array[random_index];
}
```

Loop 1 benefits from spatial locality вЂ” each element is adjacent in memory, so cache prefetching works perfectly.

Loop 2 jumps around randomly вЂ” cache misses on nearly every access.

Same number of additions. Potentially very different execution times.

## Common Misconceptions

- RAM is the fastest memory. вЂ” **Wrong.** Registers and cache are much faster.
- Cache and RAM are the same thing. вЂ” **Wrong.** Cache is on the CPU chip; RAM is separate hardware.
- The CPU executes programs directly from SSD. вЂ” **Wrong.** Programs must first load into RAM.
- More RAM automatically means a faster CPU. вЂ” **Wrong.** Cache behaviour and algorithm locality matter far more for CPU performance.
- Registers are part of RAM. вЂ” **Wrong.** Registers are inside the CPU itself.

## Quick Revision

Memory Hierarchy exists because no single memory technology can be fast, large, and cheap simultaneously.

| Level | Speed | Capacity | Cost |
|---|---|---|---|
| Registers | Fastest | Bytes (hundreds) | Highest |
| L1 Cache | Very Fast | 32вЂ“64 KB | Very High |
| L2 Cache | Fast | 256 KB вЂ“ 1 MB | High |
| L3 Cache | Moderate | 8вЂ“32 MB | Moderate |
| RAM | Slower | 8вЂ“64 GB | Low |
| SSD | Much Slower | 256 GB вЂ“ 4 TB | Lower |
| HDD | Slowest | 1вЂ“20 TB | Lowest |

Programs execute by moving data upward through this hierarchy вЂ” from SSD into RAM, from RAM into cache, from cache into registers, and finally into the CPU.

Understanding locality of reference is essential for writing high-performance C programs.
