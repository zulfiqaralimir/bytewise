# Stack and Heap

## Lesson 17: Two Memory Regions Every C Programmer Must Know

---

By the end of this lesson you should be able to answer:

- What is the Stack?
- What is the Heap?
- Why do both exist?
- What problem does each solve?
- What happens during a function call?
- What is Stack Overflow?
- What is Dynamic Memory Allocation?
- Why are Stack and Heap important in C?
- When should I use Stack vs Heap?

---

## The Fundamental Problem

When a program runs, it needs memory for:

```text
Variables
Functions
Arrays
Objects
Dynamic Data
```

Using a single memory area for all of this would be inefficient. Operating systems therefore divide a program's memory into distinct regions.

The two most important regions are the **Stack** and the **Heap**.

---

## Memory Layout of a Running Program

```text
+------------------+
|   Code Segment   |   ← compiled instructions
+------------------+
|   Global Data    |   ← global and static variables
+------------------+
|      Heap        |
|        ↑         |   ← grows upward as memory is allocated
|        ↑         |
+------------------+
|        ↓         |   ← grows downward as functions are called
|      Stack       |
+------------------+
```

The Heap grows upward and the Stack grows downward. They share the same address space and expand toward each other.

---

## What is the Stack?

> A region of memory managed automatically by the CPU for function calls, local variables, function parameters, and return addresses.

**Think of it as:** a temporary working area that is created when a function is called and destroyed when the function returns.

---

### Real-Life Analogy

Imagine a stack of plates:

```text
Plate 3  ← most recently added
Plate 2
Plate 1  ← first added
```

You can only add to the top and remove from the top. This behaviour is called **LIFO — Last In, First Out**.

The Stack in memory works exactly the same way. The last function called is the first to finish and have its frame removed.

---

## Stack Operations

### Push

Add data to the top of the Stack.

```text
Before       After
------        ------
20       →   30
10            20
              10
```

### Pop

Remove data from the top.

```text
Before       After
------        ------
30       →   20
20            10
10
```

---

## What is a Stack Frame?

Every time a function is called, the CPU pushes a **Stack Frame** onto the Stack.

A Stack Frame contains:

| Contents | Purpose |
|---|---|
| Parameters | Values passed into the function |
| Local Variables | Variables declared inside the function |
| Return Address | Where to resume execution after the function returns |
| Saved Registers | Register values the function must preserve |

---

### Example

```c
void add(int a, int b)
{
    int sum = a + b;
}
```

Stack Frame:

```text
+----------------+
| Return Address |
+----------------+
| a              |
+----------------+
| b              |
+----------------+
| sum            |
+----------------+
```

When `add` returns, the entire frame is popped and that memory is instantly reclaimed.

---

## Function Call Lifecycle

```c
void hello()
{
    int x = 10;
}
```

```text
Call hello()
      ↓
Create Stack Frame
      ↓
Store x = 10
      ↓
Execute function body
      ↓
Function returns
      ↓
Stack Frame removed automatically
```

No manual cleanup required. The CPU handles it by moving the **Stack Pointer (SP)**.

---

## Why Is the Stack Fast?

The CPU only needs to move the **Stack Pointer** — a single register that always tracks the top of the Stack.

```text
Push → SP moves down (Stack grows downward)
Pop  → SP moves back up
```

No searching, no bookkeeping, no fragmentation. Just pointer arithmetic.

---

## Nested Function Calls

```c
main()  →  funcA()  →  funcB()
```

Stack at the point inside `funcB`:

```text
+-------------+
| funcB Frame |  ← top (most recent)
+-------------+
| funcA Frame |
+-------------+
| main  Frame |  ← bottom (first)
+-------------+
```

As each function returns, its frame is popped: `funcB` first, then `funcA`, then `main`.

---

## Stack Overflow

The Stack has a fixed size (typically 1 MB–8 MB depending on the OS and settings).

If a program calls functions recursively without stopping:

```c
void func()
{
    func();   /* infinite recursion */
}
```

Each call creates a new Stack Frame. Eventually the Stack runs out of space:

```text
Stack Full  →  Stack Overflow  →  Program crashes
```

This is also why allocating enormous arrays as local variables is dangerous:

```c
int arr[10000000];   /* likely causes Stack Overflow */
```

---

## What is the Heap?

> A region of memory used for **dynamic memory allocation** — memory that is requested and released manually at runtime.

**Think of it as:** memory you ask for explicitly when you need it, and must return explicitly when you are done.

---

## Why the Heap Exists

Consider this problem:

```text
User enters the number of students at runtime.
Could be 10, 100, or 100,000.
```

You cannot know the size in advance, so you cannot declare a fixed-size array on the Stack. The Heap solves this — you request exactly as much memory as you need, when you need it.

---

## Heap Allocation in C

C provides four functions for Heap management:

| Function | Purpose |
|---|---|
| `malloc(size)` | Allocate `size` bytes; contents uninitialized |
| `calloc(n, size)` | Allocate `n` elements of `size` bytes each; zero-initialized |
| `realloc(ptr, size)` | Resize a previously allocated block |
| `free(ptr)` | Release the memory back to the Heap |

---

### Example

```c
int *arr = malloc(100 * sizeof(int));
```

```text
Heap allocator finds a free block
      ↓
Reserves 400 bytes (100 × 4)
      ↓
Returns pointer to start of block
      ↓
arr (on Stack) holds that address
```

When finished:

```c
free(arr);
```

The Heap block is released and can be reused.

---

## Stack vs Heap — Side by Side

| Feature | Stack | Heap |
|---|---|---|
| Allocation | Automatic (by CPU) | Manual (`malloc` / `calloc`) |
| Deallocation | Automatic (on function return) | Manual (`free`) |
| Speed | Faster | Slower |
| Size | Smaller (1–8 MB typical) | Larger (limited by RAM) |
| Usage | Local variables, parameters | Dynamic / runtime-sized data |
| Risk | Stack Overflow | Memory Leaks, Dangling Pointers |

---

## Why the Heap Is Slower

Stack allocation is a single instruction — move the Stack Pointer.

Heap allocation requires the allocator to:

```text
Search free list for a suitable block
      ↓
Split or merge blocks if needed
      ↓
Mark the block as used
      ↓
Return a pointer
```

More work per allocation, and Heap memory can become fragmented over time.

---

## Stack and Heap Together

```c
int *p = malloc(sizeof(int));
```

```text
Stack                      Heap
-----                      -----
p  →  address 5000         address 5000: [integer data]
```

The **pointer lives on the Stack**. The **data it points to lives on the Heap**. This is the most common pattern in C.

---

## Common Mistakes

### 1. Forgetting `free()`

```c
int *p = malloc(1000);
/* program ends without free(p) */
```

Result: **Memory Leak** — the Heap block is never returned. In long-running programs (servers, daemons), memory leaks slowly consume all available RAM.

---

### 2. Returning a Pointer to a Local Variable

```c
int* func()
{
    int x = 10;
    return &x;   /* DANGEROUS */
}
```

`x` lives on the Stack Frame of `func`. When `func` returns, that frame is destroyed. The returned pointer now points to memory that no longer belongs to `x` — a **dangling pointer**. Accessing it is undefined behaviour.

---

### 3. Allocating Large Arrays on the Stack

```c
int arr[10000000];   /* ~40 MB on Stack */
```

Exceeds the Stack size limit and causes Stack Overflow. Large arrays belong on the Heap:

```c
int *arr = malloc(10000000 * sizeof(int));
```

---

## Visual Summary

```text
Function call
      ↓
Stack Frame created (parameters, locals, return address)
      ↓
Function executes
      ↓
Function returns → Stack Frame destroyed automatically


malloc()
      ↓
Heap block allocated
      ↓
Use the memory through a pointer
      ↓
free() → Heap block released
```

## Quick Revision

| Concept | Stack | Heap |
|---|---|---|
| What is stored | Local vars, params, return addresses | Dynamically allocated data |
| Who manages it | CPU / compiler automatically | Programmer manually |
| Speed | Faster | Slower |
| Size limit | Small (fixed) | Large (RAM-limited) |
| Data structure | LIFO | Free list / allocator |
| Common problem | Stack Overflow | Memory Leak / Dangling Pointer |
| C functions | automatic | `malloc`, `calloc`, `realloc`, `free` |
