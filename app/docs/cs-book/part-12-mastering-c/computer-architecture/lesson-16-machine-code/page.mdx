# Machine Code

## Lesson 16: The CPU's Native Language

---

By the end of this lesson you should be able to answer:

- What is Machine Code?

- Why was Machine Code invented?
- Why don't humans write programs in Machine Code?
- How does Machine Code relate to Assembly Language?
- How does Machine Code relate to C programming?
- What is an Opcode?
- What is an Operand?

---

## The Fundamental Truth

A CPU does **not** understand:

```c
printf("Hello World");
```

A CPU does **not** understand:

```python
print("Hello World")
```

A CPU does **not** understand:

```assembly
MOV R1, 5
```

A CPU understands only:

```text
0   1
```

Everything eventually becomes:

```text
10110000
00000101
```

This is called **Machine Code**.

---

## What is Machine Code?

> The lowest-level programming language understood directly by the CPU. It consists entirely of binary digits: `0` and `1`.

Think of the full translation stack:

```text
Human Language

↓

Programming Language (C, Python)

↓

Assembly Language

↓

Machine Code

↓

CPU
```

Machine Code is the CPU's native language — the bottom of that stack.

---

## Why Does the CPU Understand Only Machine Code?

Inside the CPU are billions of **transistors**.

A transistor has exactly two stable electrical states:

| State | Meaning |
|---|---|
| OFF | `0` |
| ON | `1` |

A transistor cannot represent the word `ADD`. It can only represent `0` or `1`. Therefore all CPU operations ultimately reduce to patterns of binary digits.

---

### Real-Life Analogy

Humans communicate in English, Urdu, Arabic, and hundreds of other languages.

Computers communicate in exactly one language: **Binary**.

Machine Code is that language written down as a sequence of `0`s and `1`s.

---

## Example of Machine Code

```text
10110000 00000101
```

To a human: random digits.

To the CPU: *Load the value 5 into a register*.

---

## Why Machine Code Is Difficult

Imagine writing:

```text
10110000
00000101
10110001
00000011
00000001
11000001
```

instead of:

```c
a = 5 + 3;
```

Problems with writing Machine Code directly:

- Hard to read — no meaningful names, only digits
- Hard to remember — each binary pattern has a specific meaning you must memorise
- Easy to make mistakes — one flipped bit changes the entire instruction
- Extremely difficult to debug

---

## What is an Instruction?

An instruction is a command for the CPU.

Common categories:

| Category | Examples |
|---|---|
| Arithmetic | Add, Subtract, Multiply |
| Data Movement | Load from memory, Store to memory, Move between registers |
| Comparison | Compare two values |
| Control Flow | Jump to a new location, Call a function |

Every instruction is encoded as a binary pattern in Machine Code.

---

## What is an Opcode?

**Opcode** = *Operation Code*

The opcode is the part of a machine instruction that tells the CPU **what to do**.

Example mapping (illustrative — actual encodings depend on the ISA):

| Assembly Mnemonic | Opcode (binary) | Meaning |
|---|---|---|
| `MOV` | `10110000` | Copy data into a register |
| `ADD` | `00000001` | Perform addition |
| `SUB` | `00101001` | Perform subtraction |
| `JMP` | `11101001` | Jump to address |

---

## What is an Operand?

Operands tell the CPU **what to use** — the data the instruction operates on.

```assembly
ADD R1, R2
```

| Part | Value | Role |
|---|---|---|
| Opcode | `ADD` | Perform addition |
| Operand 1 | `R1` | First input (also destination) |
| Operand 2 | `R2` | Second input |

---

## Instruction Format

Most machine instructions contain:

```text
[ Opcode ]  [ Operands ]
```

Example:

```assembly
MOV R1, 5
```

Conceptual binary layout:

```text
10110000   00000101
 Opcode      Value (5)
```

The exact format — how many bits for the opcode, how many for each operand — is defined by the ISA.

---

## Machine Code and Assembly Language

Assembly Language is a human-readable representation of Machine Code. One Assembly instruction corresponds to exactly one machine instruction.

| Assembly | Machine Code |
|---|---|
| `MOV R1, 5` | `10110000 00000101` |
| `ADD R1, R2` | `00000001 11000001` |

### The Assembler

The **Assembler** is the program that performs this translation:

```text
Assembly Language  →  Assembler  →  Machine Code
```

Engineers invented Assembly because writing `MOV` is far easier than memorising `10110000`.

---

## How the CPU Executes Machine Code

Suppose memory contains:

```text
Address   Content
100       10110000   ← opcode: MOV
101       00000101   ← operand: value 5
```

The CPU processes this through the **Fetch-Decode-Execute cycle**:

### Step 1: Fetch

The CPU reads the instruction bytes from memory into the instruction register.

```text
Memory  →  CPU
```

### Step 2: Decode

The CPU interprets the opcode to determine what operation to perform.

```text
10110000  →  MOV (load a value into a register)
```

### Step 3: Execute

The CPU carries out the operation.

```text
Load value 5  →  Register R1
```

### Repeat

```text
Fetch  →  Decode  →  Execute  →  Next Instruction
```

This cycle repeats **billions of times per second** on a modern CPU.

---

## Machine Code and Registers

```assembly
MOV R1, 10
```

Machine Code tells the CPU: store `10` in Register `R1`.

Registers are used constantly because they are orders of magnitude faster than RAM. Every arithmetic operation, comparison, and data movement goes through registers.

---

## Machine Code and Memory

```assembly
LOAD  R1, [100]   ; read memory address 100 into R1
STORE [100], R1   ; write R1 back to memory address 100
```

In Machine Code, every memory access is an explicit instruction. Nothing is hidden or automatic.

---

## Machine Code and Status Flags

```assembly
CMP R1, R2
```

Machine Code instructs the CPU to subtract `R2` from `R1` internally and update the Status Register flags:

| Flag | Meaning |
|---|---|
| ZF (Zero Flag) | Result was zero — values are equal |
| SF (Sign Flag) | Result was negative |
| CF (Carry Flag) | Unsigned overflow occurred |
| OF (Overflow Flag) | Signed overflow occurred |

Conditional jump instructions then read these flags to decide whether to branch.

---

## Machine Code and C Programming

```c
int x = 5;
```

The compiler generates Assembly:

```assembly
MOV R1, 5
```

The Assembler generates Machine Code:

```text
10110000 00000101
```

The CPU executes the binary.

---

### Full Translation Path

```text
C Program

↓  Compiler

Assembly Language

↓  Assembler

Machine Code (binary)

↓

CPU Executes
```

Every C program, every Python script, every high-level language eventually becomes Machine Code.

---

## Why C Programmers Should Understand Machine Code

Understanding Machine Code explains:

- Why certain C operations are faster than others
- How registers are used during function calls and arithmetic
- What the compiler actually produces from your source code
- How to read disassembly in a debugger (`objdump`, GDB, lldb)
- How security exploits manipulate binary instructions
- Why a binary compiled for x86-64 cannot run on ARM without recompilation

---

## Why Programmers Rarely Write Machine Code

Machine Code is:

- Hard to read — no symbolic names
- Hard to debug — a single bit error is nearly impossible to spot
- Hard to maintain — no structure, no abstraction
- CPU-specific — x86-64 machine code does not run on ARM

Modern programmers write in C, C++, Rust, or Python and let compilers produce Machine Code automatically — and usually better than a human would.

## Common Misconceptions

- Machine Code and Assembly Language are the same. — **Wrong.** Assembly is human-readable text; Machine Code is binary.
- CPUs understand C directly. — **Wrong.** C must be compiled to Assembly and then assembled to Machine Code first.
- Machine Code is portable. — **Wrong.** x86-64 machine code is meaningless to an ARM CPU.
- Every CPU uses identical Machine Code. — **Wrong.** Each ISA defines its own encoding; x86-64, ARM, RISC-V, and MIPS all use different binary formats.
- Machine Code is obsolete. — **Wrong.** Every program ever run on any CPU has passed through Machine Code.

## Quick Revision

| Term | Meaning |
|---|---|
| Machine Code | Binary instructions (`0`s and `1`s) the CPU executes directly |
| Transistor | Hardware switch with two states: OFF (`0`) and ON (`1`) |
| Opcode | The operation part of an instruction — tells the CPU *what to do* |
| Operand | The data part of an instruction — tells the CPU *what to use* |
| Assembler | Program that converts Assembly Language into Machine Code |
| Fetch-Decode-Execute | The three-step cycle the CPU repeats to run every instruction |
| ISA | Defines the binary encoding format each CPU family uses |
